### 已确认

- `R34` 中“当前总控会话未注入真实密钥”的操作级阻塞已关闭；当前会话现在可以直接执行 CloudBase 侧脚本。

- 当前页面失败形态已收口为：点击登录后仍停留在登录页，并出现 `OPERATION_FAIL / PERMISSION_DENIED`。
- 总控已补做服务端对照调用：同一环境下，管理端调用 `admin-auth.login` 返回 `code=0`、`mustResetPassword=true`、`hasToken=true`；因此函数实现、账号状态与本轮一次性凭据本身当前均可用。
- 总控已在 `__ADMIN_WORKTREE__/admin-web/scripts/smoke-admin-auth-real.sh` 补入：
  - `*-after-submit-snapshot.txt`
  - 登录失败 / 直接进工作区 / 首次改密弹层出现 的显式分支
  - `旧密码` / `新密码` 的包含匹配

### 当前推论

- 现阶段应优先把问题定位为：**CloudBase Web 侧 `callFunction` 被环境级云函数安全规则拦截**。
- 当前高概率根因：`admin-web` 现状会先做匿名 CloudBase 登录，再用 SDK `callFunction` 调 `admin-auth`；而官方文档明确云函数安全规则仅对客户端 `callFunction` 生效，且默认推荐规则常为 `"auth.loginType != 'ANONYMOUS' && auth != null"`，这会稳定拦掉匿名态调用。
- 如果继续保留这套匿名 CloudBase transport，不仅 `admin-auth`，后续 `review-admin` / `governance-admin` 的真实 transport 也会受到同类规则影响。
- 现阶段不把这次失败记成：
  - 首次改密弹层锚点问题
  - 当前总控会话无密钥
  - 纯用户名密码错误
