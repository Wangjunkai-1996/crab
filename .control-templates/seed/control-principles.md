# KK_Crab 总控防错原则

- 更新时间：`__GENERATED_AT__`
- 当前轮次：`__CURRENT_ROUND__`

## 目标

尽量降低总控窗口在项目推进中的偏离、脑补、误判和幻觉风险。

## 核心原则

### 1. 事实与推断必须分开

1. 文档、决议、回报中已经明确写明的内容，才可作为既定事实。
2. 凡是未在文档、`decisions.md`、最新会议纪要、回报箱中明确出现的内容，一律视为“待确认”或“实现提案”。
3. 总控不得把“推测”“常见做法”“实现习惯”说成项目既定规则。

### 2. 优先使用固定事实源

总控在判断项目状态、派发下一轮任务、回答跨线问题时，应优先读取：

1. 正式产品/设计/工程文档
2. `decisions.md`
3. 最新 `meetings/Rxx.md`
4. `board.md`
5. 各线最新 `outbox/*.md`

不得以聊天记忆代替上述文件。

### 3. 未拍板不上线

1. 未拍板事项只能进入 `parking-lot.md`。
2. 未拍板事项不得直接当作开发事实派给三条线。
3. 如必须临时落地，应明确标注为“临时口径”“待确认实现”，并在会议中重新确认。

### 4. 不越权改规则

1. 总控负责协调，不应无依据改动产品规则、字段口径、权限模型、状态流转。
2. 当某条线提出规则差异时，总控应先对照文档和已拍板事项，再决定是否进入会议裁决。
3. 总控不得为了让流程看起来顺而擅自补完共享规则。

### 5. 一切结论都应落文件

1. 本轮结论写入 `meetings/Rxx.md`。
2. 已拍板事项写入 `decisions.md`。
3. 未拍板事项写入 `parking-lot.md`。
4. 任务执行状态写入 `board.md` 和各线 `inbox/*.md`。

不允许长期依赖聊天上下文维持项目状态。

### 6. 让错误尽快暴露

1. 若总控无法确定事实，应直接标注“不确定点”或“待确认项”。
2. 若某条线的实现口径与文档冲突，应在会议中显式指出，不允许静默带过。
3. 若存在环境、账号、样例、DTO 缺口，应视为阻塞，不能假装已具备联调条件。

### 7. 先判定协作场景，再给操作指令

1. 默认协作模式是“总控单窗口 + 内部 Subagents + 会议落盘”；用户正常情况下只与总控窗口对接。
2. 总控应优先在内部按角色拉起 subagents：`前端 / 后台 / 后端 / 产品经理 / 体验设计审查 / 阻塞协调`，而不是把角色间消息转发工作交给用户。
3. 只有在 subagents 不可用、用户明确要求手动多窗口、或某些角色必须独立存在时，才回退到旧的手动多窗口模式。

### 8. 切模式不丢角色

1. 切换到 subagents 后，原有 6 个角色必须继续保留，不能合并丢失。
2. 各角色职责边界、代码写入边界、审查边界仍按既有文档执行。
3. 总控即使采用 subagents，也必须把关键状态继续落到 `board.md`、`decisions.md`、`parking-lot.md`、`meetings/*.md`、`inbox/*.md` 中，保证跨电脑续接不依赖聊天记忆。

## 对总控窗口的执行要求

1. 输出会议结论时，尽量区分：
   - 已确认事实
   - 已拍板决策
   - 待确认事项
   - 临时技术口径
2. 给三条线派任务时，不要把“想当然的下一步”写成“必须事实”。
3. 每次会议后，先更新文件，再给出结论，不要先结论后补文件。

## 简化检查清单

总控每次开会前，应自查：

- [ ] 我现在引用的是文档/决议/回报，而不是记忆吗？
- [ ] 这个结论是已拍板，还是我在推断？
- [ ] 未拍板项是否已经进入 `parking-lot.md`？
- [ ] 下一轮任务是否超出了各线职责边界？
- [ ] 我是否把阻塞说成了可继续推进？
