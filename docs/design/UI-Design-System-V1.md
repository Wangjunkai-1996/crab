## 2. UI 总纲

### 2.1 设计定位

多米通告 V1 的视觉方向固定为“清爽可信、带编辑感的独立移动产品感”。

这意味着：

1. 页面看起来应像一款成熟的垂直移动产品，而不是微信工具页或后台页。
2. 视觉要比普通小程序更有品牌感，但不能靠重 Banner、重插画和夸张营销语言堆砌。
3. 所有设计都要服务于三个判断动作：
   - 值不值得报名
   - 要不要发布
   - 当前进展到哪一步
4. 视觉记忆点主要通过标题层级、深浅蓝叠层、纸感表面和工作台式布局建立，而不是通过运营物料堆出来。

### 2.2 品牌气质

- 可信：像一张被整理过的合作工作台，而不是信息杂货页
- 克制：装饰服从信息，不靠大面积特效制造“高级感”
- 现代：保留移动 App 的沉浸层级，并加入更强的编辑感构图
- 轻商务：对外像真实合作平台，不像社区娱乐产品
- 精致：卡片、状态、表单和底部栏都有一致的材质语言与细节完成度

### 2.3 视觉关键词

- 编辑感标题
- 深浅蓝叠层
- 冷静信号光
- 纸感高亮表面
- 精致卡片与工作台模块
- 高信息密度但仍可呼吸
- 稳定的状态标签系统
- 首屏即能做判断

### 2.4 明确避免

- 避免接近微信默认原生组件拼装感
- 避免紫色系泛科技风
- 避免大面积运营 Banner 稀释列表效率
- 避免卡通吉祥物式插图
- 避免每个页面单独定义一套按钮、标签和卡片风格

### 2.5 视觉升级锚点

以下 4 个页面是本轮 UI 审美定稿的风格锚点，后续全量页面必须沿同一套视觉语言展开：

1. 广场首页：要像产品封面页，第一眼就建立“真实合作匹配平台”的印象
2. 通告详情页：要像决策面板，首屏直接完成报名判断
3. 发布页：要像一份被拆好的专业表单，轻但不随意
4. 我的页：要像个人合作工作台，而不是入口集合页

## 3. 设计 Token 摘要

完整 token 以 [UI-Design-Tokens-V1.json](UI-Design-Tokens-V1.json) 为准，本节仅展示关键设计意图。

### 3.1 核心色彩

| 角色 | 色值 | 用途 |
| --- | --- | --- |
| Brand Primary | `#2F6BFF` | 主按钮、主选中态、品牌高亮 |
| Secondary Teal | `#12B3A8` | 次重点信息、信任辅助、完成态强调 |
| Accent Amber | `#FFB547` | 预算、亮点标签、提醒信息 |
| Danger Rose | `#E35D6A` | 风险、举报、拒绝、下架等危险态 |
| Page Background | `#F5F7FB` | 页面背景主色 |
| Soft Surface | `#EEF3F8` | 分层背景、辅助卡底 |
| Card White | `#FFFFFF` | 主卡片、弹层、表单区 |
| Text Primary | `#152033` | 主正文 |
| Text Secondary | `#5C6980` | 次级说明 |
| Text Tertiary | `#8A96A8` | 占位、弱提示 |

### 3.2 渐层与表面

1. 页面背景不再是单层平涂，允许使用 `#F5F7FB -> #EEF3F8` 为基底，再叠加极轻的品牌环境光
2. 首页、详情页和我的页头部采用更深的品牌渐层，从深墨蓝过渡到品牌蓝，并根据场景叠加少量青绿色辅助光
3. 卡片默认使用“白纸感高亮表面”，通过高光描边、浅灰结构边和柔和阴影共同建立层级
4. 底部固定操作栏继续使用磨砂白底，但上边界必须清晰，避免像悬浮 toast
5. 广告位、空状态、规则提醒等辅助模块也必须继承同一表面语言，不允许掉回朴素后台盒子感

### 3.3 字体体系

| 层级 | 建议字号 | 用途 |
| --- | --- | --- |
| Display | `56rpx` | 首页标题、成功页主文案 |
| Heading 1 | `40rpx` | 页面主标题 |
| Heading 2 | `34rpx` | 卡片标题、模块标题 |
| Heading 3 | `30rpx` | 二级模块标题 |
| Body M | `28rpx` | 正文默认 |
| Body S | `26rpx` | 辅助正文 |
| Caption | `24rpx` | 标签、副标题、说明 |
| Meta | `22rpx` | 时间、状态补充、字段说明 |

字体策略：

1. Display 和页面主标题使用更有识别度的无衬线风格，建议优先 `"HarmonyOS Sans SC"` 或 `"SF Pro Display"` 风格，并使用更紧的字距。
2. 正文使用高可读中文无衬线，兼容 `"PingFang SC"`、`"Noto Sans SC"`。
3. 数字、预算、时间、状态需要使用独立数字字重策略，形成扫读锚点。
4. Kicker、状态说明、导航短标题允许使用更宽松字距，形成类似编辑标题的节奏感。

### 3.4 间距与栅格

基础节奏统一为：

- `8rpx`
- `12rpx`
- `16rpx`
- `20rpx`
- `24rpx`
- `32rpx`
- `40rpx`
- `48rpx`

页面规则：

1. 页面左右安全留白统一 `32rpx`
2. 大标题上方留白大于普通列表页，建议 `40rpx` 到 `48rpx`
3. 卡片内部垂直节奏优先使用 `16rpx / 20rpx / 24rpx`
4. 长表单禁止直接堆叠输入框，必须按卡片分组

### 3.5 圆角、描边、阴影

| Token | 建议值 | 用途 |
| --- | --- | --- |
| Card Radius | `28rpx` | 主要信息卡片 |
| Sheet Radius | `36rpx` | 底部抽屉、反馈弹层 |
| Pill Radius | `999rpx` | 标签、筛选胶囊、按钮 |
| Border Light | `1px rgba(255,255,255,0.72)` | 卡片高光描边 |
| Shadow Soft | `0 20rpx 56rpx rgba(25, 61, 118, 0.12)` | 主卡片 |
| Shadow Float | `0 28rpx 84rpx rgba(16, 38, 78, 0.18)` | 浮层、重点卡 |

补充说明：

1. V1 卡片边界不建议只靠单条灰线，应使用“高光描边 + 结构边 + 阴影”组合来建立完成度。
2. 首页 Hero、详情 Hero、工作台 Hero 可在深色渐层中加入轻微白色雾化高光，形成独立产品感。

### 3.6 动效系统

| 场景 | 时长 | 说明 |
| --- | --- | --- |
| 轻点击反馈 | `140ms` | 按钮、chip、列表单元 |
| 模块切换 | `180ms` | Tab、筛选选中、状态切换 |
| 页面进入 | `240ms` | 新页面进入、结果态出现 |
| 抽屉与反馈弹层 | `320ms` | 底部 sheet、反馈成功层 |

动效原则：

1. 像 App 一样顺滑，但不浮夸。
2. 优先位移 + 透明度，而不是缩放弹跳。
3. 固定 CTA、筛选抽屉、卡片悬浮态必须使用统一过渡参数。

## 4. 核心组件规范

### 4.1 品牌标题导航

结构：

- 页面标题
- 页面副文案或摘要
- 可选右上操作

规则：

1. 广场、我的、发布等首页级页面允许更大标题样式。
2. 非首页级页面使用标准导航栏，高度更紧凑。
3. 标题区可叠加轻光晕背景，但不能影响可读性。

### 4.2 底部 TabBar

结构：

- 图标
- 文案
- 选中态底色或光晕

规则：

1. 允许更明显的选中态，区别于微信默认灰蓝切换。
2. 背景建议使用磨砂白或柔和纯白卡底。
3. 点击区域不小于 `88rpx` 高。

### 4.3 通告卡片

标准信息顺序：

1. 标题
2. 平台标签、合作领域标签、特征标签
3. 预算与合作形式
4. 城市与发布时间
5. 发布方名称与资料完整度
6. 状态标签

规则：

1. 标题最多 2 行。
2. 预算必须成为卡片视觉焦点之一。
3. 状态异常时优先露出状态标签。
4. 信息卡与操作卡分层，列表页以信息卡为主，管理页允许叠加操作按钮区。

### 4.4 筛选 Chip 与筛选抽屉

结构：

- 单行横向 chip
- 选中态
- 底部抽屉
- 抽屉内分组选项

规则：

1. chip 高度统一 `64rpx`
2. 选中态使用品牌主色浅底，不使用纯色硬填满
3. 抽屉内部优先使用两列或流式选项，不使用密集列表
4. 筛选项数量再多，也不允许让首页首屏变成筛选面板

### 4.5 状态标签系统

状态标签必须共用一套颜色语义：

| 状态 | 语义 |
| --- | --- |
| 待审核 | 冷静处理中 |
| 进行中 | 正常进行 |
| 已截止 | 时间结束 |
| 已关闭 | 主动结束 |
| 已下架 | 平台处理 |
| 驳回待修改 | 需用户处理 |

规则：

1. 不同页面不允许同一状态使用不同色值。
2. 状态标签必须带底纹，不能只用纯文字。
3. 报名状态和通告状态可共用同一底层语义体系，但文案保持业务准确。

### 4.6 底部固定操作栏

适用页面：

- 通告详情页
- 发布页
- 报名页
- 达人名片页
- 举报页
- 意见反馈页

规则：

1. 默认固定在底部安全区上方。
2. 允许一主一次双按钮布局。
3. 操作栏要与内容区明显分层，建议使用磨砂白底与顶部分隔线。

### 4.7 分组表单卡片

结构：

- 卡片标题
- 简短说明
- 输入项列表
- 可选提示或校验信息

规则：

1. 表单按业务语义分组，不按字段类型分组。
2. 每组输入项控制在 3 到 6 项之间。
3. 错误信息贴近字段展示，不能只在顶部统一报错。
4. 首次补资料卡需要与普通表单卡有轻微视觉区别，提示这是前置补全。

### 4.8 消息单元

结构：

- 图标或状态点
- 标题
- 摘要
- 时间
- 未读标识

规则：

1. 消息中心整体更像“通知收件箱”，不使用聊天气泡样式。
2. 审核通知、报名通知、系统通知建议提供轻度类型区分。
3. 未读不依赖纯颜色，必须同时有点位或角标。

### 4.9 空状态模块

结构：

- 产品化几何插图
- 一句话说明
- 一个主操作

规则：

1. 插图风格与整体 App 视觉一致，不使用微信原生空状态图感。
2. 文案需鼓励下一步动作，而不是只描述“暂无数据”。
3. 不同业务页可复用同一风格，但文案必须针对场景。

### 4.10 系统反馈弹层

适用场景：

- 提交成功
- 举报成功
- 资料保存成功
- 审核驳回说明

规则：

1. 使用底部反馈弹层或成功卡，而不是简单 toast 结束。
2. 成功反馈需包含“接下来会发生什么”。
3. 驳回反馈需明确原因和下一步动作入口。

## 5. 页面高保真要求

### 5.1 广场模块

#### 广场首页

- 首屏结构：大标题区、搜索框、筛选 chip、精选信息流首卡
- 视觉重点：搜索与首条通告卡片
- 状态要求：默认加载、有数据、空状态、筛选无结果、信息流广告占位

#### 搜索结果页

- 首屏结构：返回、搜索输入、历史词或热词、筛选条、结果列表
- 视觉重点：输入框与结果切换关系
- 状态要求：未搜索、已搜索、有结果、无结果

#### 通告详情页

- 首屏结构：状态区、标题、预算、平台、城市、合作形式、底部 CTA
- 视觉重点：让达人首屏内做出是否报名判断
- 状态要求：进行中、已截止、已关闭、已下架、游客态提示

### 5.2 发布方模块

#### 发布页

- 首屏结构：大标题、资料补全卡、分组表单、固定提交栏
- 视觉重点：减少填写压迫感
- 状态要求：首次补资料、正常发布、校验失败、提交中

#### 发布成功页

- 首屏结构：成功图形、主提示、审核说明、后续动作按钮
- 视觉重点：让用户知道“已经进入审核”
- 状态要求：正常提交成功、补充资料后重新提交成功

#### 我的通告列表

- 首屏结构：统计摘要卡、状态筛选、通告列表
- 视觉重点：状态概览与效率管理
- 状态要求：全部、待审核、进行中、已关闭、空状态

#### 通告编辑页

- 首屏结构：页面标题、当前状态标签、历史提示、可编辑分组表单
- 视觉重点：让用户知道这是“修改后重提”而不是重新创建
- 状态要求：驳回修改、补充资料、正常编辑

#### 报名管理页

- 首屏结构：顶部摘要、报名状态过滤、达人卡片列表
- 视觉重点：达人名片摘要与处理动作
- 状态要求：有报名、空状态、已处理状态混排

### 5.3 达人模块

#### 报名页

- 首屏结构：通告摘要卡、达人名片快照、补充字段、固定提交栏
- 视觉重点：让达人明确自己在报什么、将提交什么信息
- 状态要求：首次补名片、已补全直接报名、提交成功、校验失败

#### 我的报名列表

- 首屏结构：状态切换、报名卡片列表
- 视觉重点：一眼看懂每条报名进展
- 状态要求：全部、已查看、待联系、已沟通、未入选、空状态

#### 报名详情页

- 首屏结构：通告摘要、当前进展、时间线、联系方式释放说明
- 视觉重点：当前状态和下一步动作
- 状态要求：已报名、已查看、待联系、已沟通、未入选、已完成

#### 达人名片页

- 首屏结构：名片预览、资料分组表单、作品图上传区、固定保存栏
- 视觉重点：让用户感觉是在维护一张真正会对外展示的名片
- 状态要求：首次创建、已存在编辑、资料不完整、保存成功

### 5.4 我的与治理模块

#### 消息中心

- 首屏结构：分段过滤、通知列表、未读提示
- 视觉重点：像一个收件箱，而不是消息流
- 状态要求：全部、系统通知、审核通知、报名通知、空状态

#### 举报页

- 首屏结构：举报对象摘要、举报类型、补充说明、证据上传、提交栏
- 视觉重点：严肃但不压迫
- 状态要求：未填写、已填写、提交成功

#### 规则说明页

- 首屏结构：规则目录、规则卡片、重点警示卡
- 视觉重点：信息可快速扫读，避免法务文本堆叠感
- 状态要求：目录跳转、重点规则高亮

#### 我的页面

- 首屏结构：个人资料头卡、身份状态、快捷入口、记录面板
- 视觉重点：个人工作台感，而不是九宫格工具页
- 状态要求：游客态、双角色态、单角色态、限制状态提示

#### 举报记录页

- 首屏结构：处理进度摘要、举报记录列表
- 视觉重点：让用户知道平台正在处理
- 状态要求：处理中、已处理、未成立、空状态

#### 意见反馈页

- 首屏结构：反馈类型、问题描述、截图上传、可选联系方式、固定提交栏
- 视觉重点：简洁、友好、可快速完成
- 状态要求：未填写、可提交、提交成功

### 5.5 必须补齐的定稿状态页

以下状态页必须作为本轮 UI 定稿评审包的一部分，不可只停留在文字说明：

1. 我的页游客态
2. 我的页双角色强调态
3. 广场筛选抽屉展开态
4. 发布页校验失败态
5. 通告详情页游客态 CTA
6. 报名详情页联系方式释放态

说明：

1. 上述状态页需在高保真预览中可被直接查看，不允许仅靠开发脑补。
2. 若未来新增业务状态，不得破坏已确定的状态语义体系和底部操作规则。

## 6. 交互规则

### 6.1 页面进入

1. 页面进入使用轻上移 + 透明度过渡。
2. 首屏大标题区和首张卡片可做轻微错峰进入，增强产品感。
3. 不在普通列表滚动中加入过多动效。

### 6.2 列表与卡片

1. 卡片按下态通过轻微下沉和阴影变化反馈。
2. 下拉刷新只保留简洁加载反馈，不做花哨插画。
3. 广场首页列表滚动时，标题区可轻微压缩，但不能影响搜索入口。

### 6.3 筛选与抽屉

1. 筛选面板统一从底部上滑进入。
2. 同屏最多只有一个筛选面板处于激活状态。
3. 筛选确认后，页面应保留用户当前已选条件回显。

### 6.4 表单

1. 焦点进入输入项时，当前卡片应高亮边框。
2. 错误提示使用就近文本 + 危险态标签，不只在顶部显示。
3. 资料预填写时，必须让用户知道哪些字段来自已有资料。

### 6.5 成功与异常反馈

1. 成功反馈要告诉用户下一步会发生什么。
2. 驳回与异常反馈要配套明确可执行动作，如“重新编辑”“补充资料”。
3. 删除、关闭、撤回等高风险行为需二次确认。

### 6.6 身份与入口切换定稿

1. V1 不做全局“角色模式切换”，权限始终由用户是否具备发布方资料或达人名片决定，而不是由当前选中的身份标签决定。
2. TabBar 的“发布”始终进入发布页；若用户尚未完成发布方资料，则在发布页顶部展示资料补全卡，补全后继续当前发布流程。
3. 通告详情页的主按钮遵循固定文案策略：
   - 游客态用户：`完善达人名片后报名`
   - 已具备达人名片且未报名：`立即报名`
   - 已报名当前通告：`查看我的报名`
   - 发布方本人查看自己的通告：`查看报名`
4. “我的”页始终展示 `我的通告`、`我的报名`、`达人名片`、`消息中心` 四个核心入口；若当前能力未开通，则展示锁定态说明和补全资料 CTA，不做入口隐藏。
5. 双角色用户在“我的”页顶部展示双标签，并允许切换“本次优先视角”；该切换只影响头部摘要和快捷入口排序，不改变真实权限。
6. 双角色用户默认记忆上次在“我的”页选择的优先视角，避免每次进入都重新选择。

### 6.7 筛选器定稿

1. 广场页与搜索结果页共享同一套“发现筛选条件”；用户在任一页面修改筛选后，返回另一页面时应保留当前已选条件。
2. V1 中 `平台`、`城市`、`合作领域`、`合作形式`、`排序` 均为单选。
3. `预算` 筛选抽屉包含 `结算方式` 和 `预算区间` 两组单选项；V1 不做多选预算组合。
4. 所有筛选抽屉底部固定 `重置` 和 `确定` 两个动作；V1 不做选中即生效的实时筛选。
5. 默认排序固定为 `最新发布`，本轮不引入“智能推荐”“热度优先”等复杂排序语义。
6. 筛选项被选中后，chip 文案必须回显所选结果，例如 `预算·500-1000`，而不是继续显示通用标题。
7. 当筛选后无结果时，页面保留已选条件，并提供 `清空筛选` 主操作，不自动帮用户回退条件。

### 6.8 表单字段与校验定稿

以下规则用于锁定发布、报名、名片、举报和反馈页面的 UI 展示方式，避免开发阶段再次讨论控件与错误态。

#### 发布方资料卡

- `身份类型`：单选，必填
- `对外展示名称`：单行输入，必填，最多 20 字
- `所在城市`：选择器，必填
- `联系方式类型`：单选，必填
- `联系方式内容`：单行输入，必填，最多 40 字
- `简介`：多行输入，选填，最多 80 字

#### 发布页表单

- `通告标题`：单行输入，必填，最多 28 字
- `品牌或门店名称`：单行输入，选填，最多 20 字
- `合作平台` / `合作领域` / `合作形式`：单选，必填
- `城市`：选择器，必填
- `结算方式`：单选，必填
- `预算区间`：当结算方式为固定报价或可议价时必填，否则自动置灰并展示“不适用”
- `招募人数`：数字输入，选填
- `报名截止时间`：日期时间选择，必填
- `达人要求`：多行输入，必填，最多 200 字
- `合作说明`：多行输入，必填，最多 500 字
- `补充图片`：选填，最多 6 张

#### 报名页表单

- `自我介绍`：多行输入，必填，最多 200 字
- `可提供的合作内容`：多行输入，必填，最多 200 字
- `期望报价或合作条件`：多行输入，选填，最多 120 字
- `补充案例截图`：选填，最多 3 张
- `联系方式覆盖`：默认不单独暴露，仅在需要时允许用户覆盖本次报名联系方式

#### 达人名片页表单

- `昵称`：单行输入，必填，最多 20 字
- `所在城市`：选择器，必填
- `性别`：选填
- `擅长平台` / `擅长领域` / `粉丝量级`：单选，必填
- `账号名称`：单行输入，选填，最多 30 字
- `账号链接或 ID`：单行输入，选填，最多 100 字
- `代表作品截图`：必填，至少 1 张，最多 6 张
- `合作案例说明`：多行输入，选填，最多 200 字
- `常驻城市`：选择器，选填
- `联系方式类型` / `联系方式内容`：必填，联系方式内容最多 40 字

#### 举报与反馈表单

- 举报页 `举报原因` 必填，`补充说明` 最多 200 字，`凭证图片` 最多 6 张
- 反馈页 `反馈类型` 必填，`问题描述` 最多 300 字且必填，`截图` 最多 4 张，`联系方式` 选填且最多 40 字

#### 通用校验规则

1. 主按钮在必填项未完成前保持禁用态，不使用“点了再全页报错”的处理方式。
2. 错误提示必须显示在字段下方，并配套危险态描边。
3. 资料预填写字段必须显示“已从资料带入”或同等级提示，避免用户误以为是本次手动输入。
4. 上传区在未上传前需展示数量限制与用途提示。

### 6.9 品牌与视觉锁定

1. V1 正式 UI 主色锁定为 `#2F6BFF`，辅助色锁定为 `#12B3A8`，强调色锁定为 `#FFB547`，危险色锁定为 `#E35D6A`。
2. V1 视觉标识默认使用文字标识“多米通告”，当前不强制依赖图形 Logo 才可进入开发。
3. 启动页短文案锁定为 `真实合作，更快匹配`。
4. 一句话介绍默认使用：`多米通告，帮助有合作需求的人快速发通告，帮助中小达人高效找合作。`
5. 分享图、主体信息、客服信息仍可保留占位，但不得影响页面配色、标题样式和组件结构的开发落地。

### 6.10 定稿边界说明

1. 本轮 UI 定稿以 [UI-High-Fidelity-Prototype-V1.html](UI-High-Fidelity-Prototype-V1.html) 与本规范共同作为评审依据。
2. 只要不触碰信息架构、角色权限、主色系统、核心组件样式和上述交互决策，后续实现中的像素级微调可视为开发还原优化，不视为重新开 UI 稿。
3. 若未来新增广告、会员或认证模块，必须另开版本，不得回写改变 V1 已锁定页面骨架。

## 7. 小程序实现边界

1. 虽然整体气质偏独立移动产品，但保留微信小程序的基础操作习惯。
2. 不设计登录页，不设计认证前置页，不设计强制引导全屏 onboarding。
3. 所有底部 CTA、TabBar 和弹层必须考虑安全区。
4. 浮层数量要克制，避免小程序内出现连续多层模态。
5. 图标资源、插图资源和 TabBar 方案需要控制体积，避免影响小程序加载。

## 8. 可访问性与可用性

1. 正文与背景对比度至少满足可读要求，正文不低于 4.5:1。
2. 小号辅助文案只允许用于非核心信息，不能承载关键业务判断。
3. 点击热区不小于 `88rpx x 88rpx`。
4. 预算、状态、时间等信息不能只依赖颜色区分。
5. 表单中所有必填项必须有显式标记或明确字段说明。

## 9. 开发交付要求

1. 前端实现优先读取 [UI-Design-Tokens-V1.json](UI-Design-Tokens-V1.json) 中的颜色、圆角、阴影、字号和状态 token。
2. 所有列表卡片、标签和按钮优先沉淀成可复用组件，不要页面内写死多套样式。
3. 高保真预览稿 [UI-High-Fidelity-Prototype-V1.html](UI-High-Fidelity-Prototype-V1.html) 用于视觉对齐，不作为真实数据结构定义来源。
4. 业务字段、状态与可见性规则仍以 [Field-Dictionary-V1.md](../product/Field-Dictionary-V1.md) 和 [Visibility-Permissions-Matrix-V1.md](../product/Visibility-Permissions-Matrix-V1.md) 为准。

## 10. UI 验收清单

### 10.1 视觉一致性

- 是否全量页面保持同一品牌气质
- 是否避免业务页像工具页、治理页像后台页的割裂
- 是否所有状态、卡片、按钮共用统一语言

### 10.2 信息效率

- 是否首屏即可判断通告值不值得报名
- 是否列表卡片能在 3 秒内完成扫读
- 是否我的页与报名详情页能快速看懂当前进展

### 10.3 表单体验

- 是否按语义分组，而不是长列表堆叠
- 是否固定 CTA 明确
- 是否校验反馈足够贴近输入项

### 10.4 开发可还原性

- 是否已有统一 token
- 是否已有核心组件规范
- 是否所有关键页面都有明确视觉锚点

### 10.5 定稿判定

- 是否已消除“前端自行决定”的身份切换、筛选生效、表单校验和 CTA 文案分歧
- 是否已在高保真稿中补齐游客态、双角色态、筛选抽屉态、表单错误态、联系方式释放态
- 是否已锁定 V1 品牌主色与默认文案，不再出现“视觉已做但品牌未定”的冲突
- 若以上均满足，则本轮可作为 V1 UI 正式定稿版本进入前端开发
