## 2. 背景与目标

前台小程序采用“低门槛发布、免费优先、轻风控兜底”的策略，这意味着平台必须依赖一套足够轻但可执行的运营后台，来承接以下工作：

1. 审核新发布的通告
2. 处理用户举报
3. 管理违规用户和黑名单
4. 跟踪风险内容和处理结果

V1 不追求复杂的运营系统，只要求后台能完成最小可行运营闭环，确保前台内容质量可控。

## 3. 后台角色

### 3.1 角色定义

- 审核员：负责通告审核、基础举报处理
- 运营管理员：负责复杂举报、黑名单管理、规则执行
- 超级管理员：负责权限分配与规则调整

### 3.2 权限边界

| 角色 | 通告审核 | 举报处理 | 黑名单管理 | 规则配置 | 权限管理 |
| --- | --- | --- | --- | --- | --- |
| 审核员 | 可执行 | 可执行基础处理 | 不可执行 | 不可执行 | 不可执行 |
| 运营管理员 | 可执行 | 可执行 | 可执行 | 可执行基础规则 | 不可执行 |
| 超级管理员 | 可执行 | 可执行 | 可执行 | 可执行 | 可执行 |

## 4. V1 后台范围

V1 运营后台仅包含以下模块：

1. 首页工作台
2. 通告审核列表
3. 通告审核详情
4. 举报管理列表
5. 举报处理详情
6. 黑名单管理
7. 处理记录查询

V1 明确不做：

1. 复杂 BI 仪表盘
2. 自动审核规则后台化配置
3. 批量导出报表
4. 多组织权限体系
5. 工单系统

## 5. 核心业务目标

### 5.1 通告审核目标

- 新用户前 3 条通告必须进入人工审核
- 高风险通告不能直接流入广场
- 审核结果要可追踪、可回溯

### 5.2 举报处理目标

- 所有举报必须留痕
- 同一对象多次举报可聚合查看
- 举报结论要能反向影响通告或账号状态

### 5.3 黑名单目标

- 对重复违规对象做限制发布或限制报名
- 对高风险对象支持立即封禁
- 对所有处罚保留原因和生效时间

## 6. 首页工作台

### 6.1 页面目标

让审核和运营人员进入后台后，第一眼看到待处理事项。

### 6.2 页面模块

1. 待审核通告数
2. 待处理举报数
3. 今日新增通告数
4. 今日通过数 / 驳回数
5. 今日新增黑名单数
6. 最近 10 条待优先处理内容

### 6.3 快捷入口

- 去审核通告
- 去处理举报
- 去查看黑名单

### 6.4 优先级规则

工作台应优先展示以下内容：

1. 命中高风险规则的通告
2. 被多次举报的对象
3. 待处理时间超过 SLA 的任务

## 7. 通告审核列表

### 7.1 页面目标

帮助审核员高效处理待审核通告。

### 7.2 列表字段

- 通告 ID
- 通告标题
- 发布方名称
- 身份类型
- 城市
- 合作平台
- 合作形式
- 结算方式
- 预算区间
- 提交时间
- 当前通告状态
- 当前审核任务阶段
- 风险标签
- 历史发布数
- 历史违规数

### 7.3 筛选条件

- 审核状态
- 提交时间
- 身份类型
- 城市
- 平台
- 结算方式
- 风险等级
- 是否新用户

### 7.4 审核动作

- 通过
- 驳回
- 需补充资料
- 转人工复核
- 直接下架

### 7.5 列表规则

1. 默认按提交时间倒序排列
2. 高风险内容置顶
3. 支持单条审核，不要求 V1 支持批量审核

### 7.6 审核任务规则

1. 用户每次提交通告审核时，系统应创建一条新的审核任务。
2. “转人工复核”应结束当前任务，并自动创建一条新的人工复核任务。
3. 当通告被要求补充资料或被驳回后，用户再次提交时应创建新的复审任务。
4. 后台列表中应同时可见“通告业务状态”和“当前审核任务阶段”，避免混用。

## 8. 通告审核详情

### 8.1 页面目标

在一个页面里完成审核判断与操作执行。

### 8.2 页面结构

1. 通告基础信息
2. 发布方信息
3. 风险提示区
4. 历史记录区
5. 审核操作区

### 8.3 通告基础信息

- 标题
- 合作平台
- 合作形式
- 城市
- 结算方式与预算区间
- 招募人数
- 报名截止时间
- 达人要求
- 合作说明
- 联系方式
- 图片与截图

### 8.4 发布方信息

- 用户 ID
- 发布方名称
- 身份类型
- 资料完整度
- 历史发布数
- 历史通过数
- 历史驳回数
- 历史违规数

### 8.5 风险提示区

系统可展示以下风险提示：

- 敏感词命中
- 联系方式异常
- 内容重复率高
- 高频发布
- 多次被举报

### 8.6 审核操作说明

#### 通过

适用于内容完整、风险低、符合平台主题的通告。

#### 驳回

适用于信息缺失、描述不清、明显低质但可修改的通告。

#### 需补充资料

适用于主体基本可信，但信息不足以判断是否适合上架。

#### 转人工复核

适用于规则无法直接判定、疑似高风险但证据不足的通告。

补充规则：

1. 当前审核任务结束，结果记为“转人工复核”。
2. 通告业务状态保持“待审核”。
3. 系统自动创建新的人工复核任务进入复核队列。

#### 直接下架

适用于违法违规、欺诈、恶意导流或重复严重违规的通告。

### 8.7 审核结果字段

后台提交审核结果时必须填写：

- 处理动作
- 处理原因分类
- 处理备注
- 是否通知用户
- 下一队列去向（仅转人工复核时）

### 9.1 页面目标

帮助运营快速识别高风险对象和集中投诉问题。

### 9.2 列表字段

- 举报 ID
- 举报对象类型
- 举报对象 ID
- 举报原因
- 被举报对象名称
- 举报人 ID
- 举报时间
- 当前处理状态
- 累计举报次数
- 是否命中高风险

### 9.3 筛选条件

- 对象类型
- 举报原因
- 处理状态
- 举报时间
- 风险等级

### 9.4 列表能力

1. 支持按对象聚合查看举报
2. 支持查看同一对象历史处理记录
3. 支持从举报列表直接跳转到通告详情或用户详情

## 10. 举报处理详情

### 10.1 页面目标

让运营对举报做出明确结论，并联动处罚动作。

### 10.2 页面结构

1. 举报摘要
2. 被举报内容详情
3. 历史举报记录
4. 运营处理区

### 10.3 处理动作

- 举报不成立
- 警告用户
- 下架通告
- 限制发布
- 限制报名
- 加入黑名单

### 10.4 处理信息必填项

- 结论
- 处理原因
- 备注
- 生效时长
- 是否通知相关用户

### 10.5 联动规则

1. 若举报成立且对象是通告，系统应支持一键下架
2. 若举报成立且对象是账号，系统应支持一键限制发布或报名
3. 若同一对象多次举报成立，系统应建议进入黑名单审核

## 11. 黑名单管理

### 11.1 页面目标

集中管理已处罚对象，防止重复违规继续扩散。

### 11.2 列表字段

- 用户 ID
- 用户昵称或发布方名称
- 限制类型
- 生效时间
- 到期时间
- 处罚原因
- 处罚执行人
- 当前状态

### 11.3 限制类型

- 禁止发布
- 禁止报名
- 全量封禁
- 仅观察名单

### 11.4 操作能力

- 新增处罚
- 延长处罚
- 提前解除
- 查看历史处罚记录

### 11.5 业务规则

1. 黑名单处罚必须保留原因和证据说明
2. 处罚解除后不删除历史记录
3. 观察名单不直接限制行为，但会提升审核优先级

### 12.3 记录字段

- 记录 ID
- 对象类型
- 对象 ID

## 13. 后台与前台联动规则

### 13.1 审核联动

1. 审核通过后，通告进入广场展示
2. 驳回后，前台“我的通告”显示驳回原因
3. 需补充资料后，前台“我的通告”显示补资料原因，用户需修改后再次提交
4. 转人工复核后，前台仍显示“待审核”，后台关闭当前任务并生成新的复核任务
5. 下架后，详情页不可继续报名

### 13.2 举报联动

1. 举报成立后，相关通告或用户状态同步更新
2. 举报处理结果进入消息通知
3. 被处罚用户在前台看到限制说明

### 13.3 黑名单联动

1. 被限制发布的用户不能新建通告
2. 被限制报名的用户不能提交报名
3. 被全量封禁的用户只能浏览公开内容

## 14. 后台 SLA 建议

### 14.1 审核 SLA

- 新通告：4 小时内处理
- 高风险通告：1 小时内优先处理

### 14.2 举报 SLA

- 普通举报：24 小时内处理
- 疑似欺诈或骚扰：4 小时内处理

### 14.3 黑名单处理 SLA

- 高风险处罚：即时生效
- 一般处罚：处理后 10 分钟内生效

## 15. V1 验收标准

1. 审核员能独立完成通告审核闭环
2. 运营能独立完成举报处理闭环
3. 黑名单处罚能联动前台限制行为
4. 所有处理动作可追溯到具体操作人和时间
5. 后台不依赖复杂配置即可支撑冷启动阶段运营
