## 2. 项目背景

当前市面上的达人合作平台普遍存在三个问题：

1. 发布门槛高。大量需求发布方并非严格意义上的企业商家，还包括个人 PR、门店负责人、代运营、代理商等，他们往往没有完整企业认证材料，导致真实需求无法顺利发布。
2. 产品链路重。认证、会员、增值能力往往前置，导致供给侧和需求侧在早期都难以形成足够活跃度。
3. 中小博主体验差。平台更偏向头部达人和高预算合作，中小达人很难找到低门槛、真实、可快速报名的合作机会。

因此，本产品 V1 不追求复制“强平台、强认证、强会员”的模式，而是采用“低门槛发布、全链路免费、轻风控兜底”的冷启动路径，优先解决最核心的问题：让真实需求可以快速流通，让达人可以快速找到并报名合作。

## 3. 产品定位

### 3.1 一句话定位

帮助有推广需求的人快速发布合作招募，帮助中小达人高效找到真实合作机会。

### 3.2 核心价值

- 对发布方：无需复杂认证，也能快速发出合作需求。
- 对达人：免费浏览和报名，快速接触真实合作机会。
- 对平台：先跑通撮合链路，再通过广告而非会员收费进行后续商业化。

### 3.3 产品原则

1. 低门槛优先：不做强制企业认证，不设计独立注册流程。
2. 简单链路优先：聚焦“发布、浏览、报名、查看结果”四个关键动作。
3. 免费优先：V1 不做会员收费，不做付费解锁。
4. 轻风控优先：不通过重审核阻挡发布，而是通过审核、举报、黑名单、限流等手段控制风险。
5. 冷启动优先：优先服务中小发布方与中小达人，先建立活跃供需池。

## 4. 目标用户

### 4.1 发布方

发布方不是狭义商家，而是所有有达人合作需求的人，包含但不限于：

- 本地门店负责人
- 品牌市场或新媒体运营
- 个人 PR
- 代理商
- 代运营团队
- 创业品牌主理人

发布方核心诉求：

- 快速发布合作需求
- 快速收到报名
- 低成本筛选合适达人
- 避免复杂认证流程

### 4.2 达人方

达人方指希望接单的内容创作者，重点覆盖中小达人：

- 小红书博主
- 抖音达人
- 视频号创作者
- 本地探店达人
- 垂类内容创作者

达人核心诉求：

- 免费发现合作机会
- 看到信息完整、相对真实的需求
- 快速报名，不被复杂流程打断
- 更透明地判断是否值得投递

### 4.3 平台运营方

平台运营负责：

- 审核通告
- 处理举报
- 管理黑名单
- 管理广告位与投放节奏
- 监控供需质量

## 5. 版本目标与非目标

### 5.1 V1 目标

1. 跑通需求发布到达人报名的最短闭环。
2. 在无强认证前提下，建立基础可信度与风控能力。
3. 保持核心功能免费，降低双边使用门槛。
4. 建立可持续迭代的基础数据结构与页面骨架。

### 5.2 V1 非目标

1. 不做企业认证体系。
2. 不做会员体系和付费增值功能。
3. 不做担保交易、平台结算、站内支付。
4. 不做自动抓取全平台达人数据。
5. 不做复杂推荐算法。
6. 不做站内即时聊天系统。

## 6. 核心成功指标

### 6.1 北极星指标

- 周有效报名通告数

### 6.2 关键业务指标

- 通告发布成功率
- 通告审核通过率
- 通告发布后 48 小时内获得至少 1 次报名的比例
- 达人报名转化率
- 达人重复报名率
- 举报率
- 审核打回率
- 违规下架率

### 6.3 质量指标

- 通告平均审核时长小于 4 小时
- 通告详情页缺失关键信息比例低于 15%
- 虚假或低质通告举报成立率低于 5%

## 7. 产品范围

### 7.1 核心业务闭环

1. 发布方进入小程序
2. 选择身份并完善最小资料
3. 发布合作通告
4. 平台进行轻审核
5. 通告进入广场
6. 达人浏览、筛选、查看详情
7. 达人提交报名
8. 发布方查看报名列表并筛选意向达人
9. 平台通过站内通知反馈关键状态

### 7.2 小程序信息架构

建议采用 4 个一级导航：

1. 广场
2. 发布
3. 消息
4. 我的

补充说明：

- 广场负责浏览、搜索、筛选和查看详情。
- 发布承担发布方的核心转化入口。
- 消息承载审核结果、报名状态、系统通知。
- 我的集中承载个人资料、我的通告、我的报名、举报记录等。

## 8. 角色与权限

### 8.1 角色定义

- 游客态用户：已进入小程序但未完成发布方资料或达人名片的用户，仅可浏览公开内容，不可发布与报名。
- 发布方：可创建、编辑、关闭通告，可查看报名列表。
- 达人：可创建达人名片，可报名通告。
- 双角色用户：可同时具备发布方和达人身份。
- 运营管理员：可审核、下架、拉黑、处理举报。

### 8.2 权限原则

1. 用户进入小程序后，系统应基于微信身份创建或识别平台用户，不设计额外注册页。
2. “游客态”仅指未完成角色资料的浏览态用户，不代表匿名未登录用户。
3. 用户首次执行“发布”或“报名”时，再补充对应最小资料。
4. 同一用户可切换身份，不强制二选一。

## 10. 核心流程

### 10.1 发布方流程

1. 进入小程序
2. 点击“发布”
3. 选择发布方身份类型
4. 补充最小资料
5. 填写通告表单
6. 提交审核
7. 审核通过后在广场展示
8. 查看报名列表
9. 标记意向、已沟通或不合适
10. 关闭通告

### 10.2 达人流程

1. 进入小程序
2. 浏览广场或搜索
3. 查看通告详情
4. 首次报名时完善达人名片
5. 提交报名
6. 接收报名状态通知
7. 在“我的报名”中查看进展
8. 在报名详情中查看联系方式释放和最终结果

### 10.3 运营流程

1. 查看待审核通告
2. 审核通过、驳回或人工复核
3. 处理用户举报
4. 下架违规通告
5. 处理黑名单与限制发布

## 11. 详细功能需求

### 11.1 登录与身份机制

#### 功能目标

降低进入门槛，不让用户在初次进入时被注册、认证流程打断。

#### 详细需求

1. 用户进入小程序后，系统应默认允许浏览广场公开通告。
2. 用户首次点击“发布”时，系统应引导其选择发布方身份类型。
3. 用户首次点击“报名”时，系统应引导其创建达人名片。
4. 用户可同时维护发布方资料和达人资料。
5. 系统不提供企业认证作为必经前置步骤。
6. 系统应为同一微信身份维护唯一平台用户档案。
7. 发布方资料与达人名片应分别独立维护，互不覆盖。
8. V1 达人侧只支持一张达人名片，不支持多套名片切换。

#### 发布方最小资料

- 身份类型：商家 / 个人 PR / 代理商 / 代运营 / 其他
- 对外展示名称
- 所在城市
- 联系方式
- 简介

#### 达人最小资料

- 昵称
- 所在城市
- 擅长平台
- 擅长领域
- 粉丝量级
- 合作案例截图
- 联系方式

#### 身份初始化规则

1. 用户首次进入小程序时，系统应先创建基础用户记录，并允许其直接浏览公开内容。
2. 用户完成发布方最小资料后，获得发布能力，但不自动生成达人名片。
3. 用户完成达人最小资料后，获得报名能力，但不影响其发布方资料状态。
4. 若账号被限制发布或限制报名，系统应仅禁用对应能力，保留浏览能力。

#### 首次补资料交互原则

1. 用户首次点击“发布”进入发布页时，系统应在页内优先展示发布方最小资料补充卡，未保存前禁用通告提交能力。
2. 发布方最小资料保存成功后，用户应停留在当前发布页继续填写通告，已输入的通告内容不得丢失。
3. 用户首次点击“报名”时，系统应跳转达人名片页补资料。
4. 达人名片保存成功后，系统应返回原报名页继续提交，已输入的报名草稿不得丢失。

#### EARS 验收要求

1. 当用户首次点击“发布”时，系统应要求其补充发布方最小资料。
2. 当用户首次点击“报名”时，系统应要求其补充达人最小资料。
3. 若用户资料未填写完成，系统应阻止提交并高亮缺失项。
4. 系统应允许同一微信身份同时拥有发布和报名能力。

### 11.2 广场首页

#### 功能目标

帮助达人快速发现可报名的合作需求，帮助发布方获得曝光。

#### 详细需求

1. 广场以信息流形式展示已审核通过的通告。
2. 每张卡片展示以下信息：
   - 标题
   - 合作平台
   - 合作领域
   - 城市
   - 预算摘要（由结算方式和预算区间生成）
   - 招募人数
   - 发布时间
   - 发布方名称
   - 资料完整度标签
3. 支持基础筛选：
   - 平台
   - 城市
   - 合作领域
   - 结算方式
   - 预算区间
   - 合作形式
4. 支持关键词搜索。
5. 支持按最新发布排序。
6. 支持展示“急招”或“高预算”标签，但不做付费置顶。

#### 交互规则

1. 下拉刷新最新通告。
2. 列表到底自动加载更多。
3. 通告已关闭或已过期后不再进入默认列表。

#### EARS 验收要求

1. 当通告审核通过时，系统应将其加入广场可展示列表。
2. 当用户设置筛选条件时，系统应仅展示满足条件的通告。
3. 若通告状态为已关闭、已过期或已下架，系统应避免继续展示在默认广场列表中。

### 11.3 搜索与筛选

#### 功能目标

提升达人发现效率，减少无效浏览。

#### 详细需求

1. 搜索支持标题关键词、品牌名、城市关键词。
2. 支持保留最近 10 条搜索历史。
3. 空结果页提供重新筛选建议。
4. 筛选项应支持组合使用。

#### EARS 验收要求

1. 当用户输入关键词搜索时，系统应返回标题、品牌名或城市匹配的通告。
2. 若搜索结果为空，系统应展示空结果态与推荐筛选条件。

### 11.4 通告详情页

#### 功能目标

在详情页完成信息判断和报名转化。

#### 详细需求

通告详情页至少展示：

- 通告标题
- 发布状态
- 合作平台
- 合作领域
- 合作形式
- 所属城市
- 结算方式
- 预算区间
- 招募人数
- 报名截止时间
- 合作说明
- 达人要求
- 发布方名称
- 发布方身份类型
- 发布方资料完整度
- 历史发布数
- 举报入口
- 报名按钮

#### 规则说明

1. 发布方联系方式在广场和详情页默认不公开展示。
2. 达人提交报名前，可看到“报名后由发布方查看并联系”的提示，但不可见发布方原始联系方式。
3. 达人报名后，其联系方式在发布方报名列表中默认脱敏；发布方点击“查看联系方式”或将状态标记为“待联系”后，系统才展示完整联系方式并记录查看时间。
4. 当报名状态进入“待联系”或“已沟通”后，对应达人可查看该通告的发布方联系方式。
5. 若通告状态非“进行中”，报名按钮应置灰并展示原因。

#### EARS 验收要求

1. 当用户进入通告详情页时，系统应展示报名所需的完整关键信息。
2. 若通告已截止、已关闭或已下架，系统应禁止继续报名。
3. 若用户发现异常信息，系统应提供举报入口。

### 11.5 发布通告

#### 功能目标

让发布方可以在最短时间内完成高质量通告发布。

#### 表单字段

- 通告标题
- 身份类型
- 品牌或门店名称
- 合作平台
- 合作领域：本地生活 / 美食餐饮 / 美妆穿搭 / 母婴亲子 / 家居生活 / 数码汽车 / 旅行户外 / 教育服务 / 其他
- 合作形式：探店 / 寄拍 / 图文 / 短视频 / 直播 / 其他
- 所在城市
- 结算方式：固定报价 / 可议价 / 置换合作 / 免费体验 / 其他
- 预算区间
- 招募人数
- 报名截止时间
- 达人要求
- 合作说明
- 联系方式
- 补充图片或截图

#### 表单规则

1. 必填字段：
   - 通告标题
   - 身份类型
   - 合作平台
   - 合作领域
   - 所在城市
   - 结算方式
   - 达人要求
   - 合作说明
   - 联系方式
2. 当结算方式为“固定报价”或“可议价”时，预算区间必填。
3. 当结算方式为“置换合作”“免费体验”或“其他”时，预算区间默认记录为“不适用”。
4. 系统应提供字段示例，帮助用户提高填写质量。
5. 系统应允许保存草稿。
6. 提交后状态进入“待审核”。
7. 审核被驳回或要求补充资料后，用户可查看原因并重新编辑提交。

#### 状态流转

- 草稿
- 待审核
- 驳回待修改
- 需补充资料
- 进行中
- 已截止
- 已关闭
- 已下架

说明：

- “审核通过”作为审核动作结果保留在后台记录中，但前台通告持久状态直接进入“进行中”。

#### EARS 验收要求

1. 当用户提交通告时，系统应校验所有必填项是否完整。
2. 若存在缺失或明显无效信息，系统应阻止提交并提示修改。
3. 当通告提交成功时，系统应将状态设置为待审核。
4. 若审核驳回或要求补充资料，系统应向用户展示处理原因并允许再次提交。
5. 若结算方式要求填写预算区间但用户未填写，系统应阻止提交。

### 11.6 审核机制

#### 功能目标

在不抬高使用门槛的前提下，控制低质和违规内容。

#### 审核策略

1. 新用户前 3 条通告默认进入人工审核。
2. 老用户可根据历史表现切换为轻审核或抽检。
3. 命中敏感词、联系方式异常、重复发布、低质量模板内容时，进入人工复核。
4. 审核结果包括：
   - 通过
   - 驳回
   - 需补充资料
   - 下架
5. “转人工复核”仅作为后台审核任务结果保留，不作为前台通告持久状态。

#### 审核标准

1. 信息是否完整
2. 是否包含虚假承诺或明显欺诈
3. 是否存在色情、违法、违规内容
4. 是否与达人合作主题明显无关
5. 是否存在恶意导流或垃圾广告

#### EARS 验收要求

1. 当新用户提交前 3 条通告时，系统应将其路由到人工审核队列。
2. 若通告命中敏感词或高风险规则，系统应避免直接上架并进入人工复核。
3. 当审核完成时，系统应向发布方发送审核结果通知。

### 11.7 达人名片

#### 功能目标

帮助达人用最少信息完成可判断的自我展示。

#### 字段设计

- 头像
- 昵称
- 所在城市
- 性别（可选）
- 擅长平台
- 擅长领域
- 粉丝量级
- 账号名称
- 账号链接或账号 ID
- 代表作品截图
- 合作案例描述
- 常驻城市
- 联系方式

#### 规则说明

1. 第一版不接入平台自动拉取数据，全部手动填写。
2. 上传截图作为辅助证明，但不展示“官方认证”字样。
3. 平台展示“资料完整度”，不展示“认证等级”。
4. V1 只支持单一达人名片，不支持多名片切换。
5. 达人擅长领域建议与通告合作领域使用同一套枚举，便于筛选和匹配。

#### EARS 验收要求

1. 当达人首次报名时，系统应引导其完成达人名片。
2. 若达人名片缺少必要信息，系统应阻止报名提交。
3. 系统应根据已填写字段数量生成资料完整度。

### 11.8 报名通告

#### 功能目标

让达人用最低成本完成一次有效报名。

#### 报名字段

- 使用当前达人名片（V1 单名片）
- 自我介绍
- 可提供的合作内容
- 期望报价或合作条件
- 补充案例截图
- 联系方式确认（默认带出达人名片联系方式，可本次修改并保存为报名快照）

#### 规则说明

1. 同一达人对同一通告仅可报名一次。
2. 达人可在通告截止前撤回报名。
3. 报名成功后进入“已报名”状态。
4. 发布方查看报名后，状态可更新为“已查看”。
5. 发布方标记沟通意向后，状态可更新为“待联系”或“已沟通”。
6. 达人联系方式在报名管理列表默认脱敏，发布方点击“查看联系方式”或标记“待联系”后才展示完整值。

#### 状态流转

- 已报名
- 已查看
- 待联系
- 已沟通
- 未入选
- 已撤回
- 已完成合作

#### EARS 验收要求

1. 当达人提交报名时，系统应校验其是否已报名过该通告。
2. 若达人已报名过该通告，系统应避免重复提交。
3. 当报名提交成功时，系统应通知发布方有新报名。
4. 当达人提交报名时，系统应保存本次联系方式快照用于后续联系展示。

### 11.9 报名管理

#### 功能目标

让发布方快速筛选出潜在合适达人。

#### 详细需求

1. 发布方可查看单个通告下的全部报名者。
2. 报名列表展示：
   - 达人昵称
   - 平台
   - 领域
   - 粉丝量级
   - 所在城市
   - 自我介绍摘要
   - 报名时间
   - 当前状态
   - 联系方式查看状态
3. 发布方可执行以下操作：
   - 标记已查看
   - 查看联系方式
   - 标记待联系
   - 标记不合适
   - 标记已完成合作
4. 第一版不提供 Excel 导出。

#### EARS 验收要求

1. 当发布方进入报名管理页时，系统应展示当前通告下的所有报名记录。
2. 当发布方更新报名状态时，系统应同步在达人侧展示最新状态。
3. 当发布方将报名标记为“待联系”时，系统应向对应达人开放发布方联系方式。

### 11.10 我的通告

#### 功能目标

为发布方提供最基础的发布后管理能力。

#### 详细需求

1. 展示通告列表及状态。
2. 支持查看每条通告的报名人数。
3. 支持编辑、关闭、重新发布。
4. 支持查看审核状态、驳回原因和补资料原因。

#### EARS 验收要求

1. 当发布方打开“我的通告”时，系统应按状态分组展示其已发布内容。
2. 若通告处于驳回待修改状态，系统应允许用户直接进入编辑流程。
3. 若通告处于需补充资料状态，系统应允许用户直接进入编辑流程并展示补资料原因。

### 11.11 我的报名

#### 功能目标

为达人提供报名记录追踪能力。

#### 详细需求

1. 展示已报名通告列表。
2. 展示每条报名的最新状态。
3. 支持进入报名详情页查看状态、报名快照和联系方式释放情况。
4. 支持撤回未处理报名。

#### EARS 验收要求

1. 当达人打开“我的报名”时，系统应展示全部历史报名记录及状态。
2. 若报名仍处于未处理状态，系统应允许达人撤回报名。

### 11.12 报名详情

#### 功能目标

为达人提供单条报名记录的完整进展查看能力，承接联系方式释放和合作结果反馈。

#### 详细需求

1. 展示对应通告摘要信息，包括标题、平台、城市、预算摘要和当前通告状态。
2. 展示当前报名状态与状态时间线。
3. 展示达人本次提交的报名内容快照，包括自我介绍、合作条件、案例截图和联系方式快照。
4. 当报名状态进入“待联系”或“已沟通”后，展示该通告的发布方联系方式。
5. 当报名尚未释放发布方联系方式时，展示“等待发布方开放联系”的说明态。
6. 当报名状态为“未入选”时，展示结果说明和返回广场按钮。
7. 当报名状态仍可撤回时，提供撤回入口。

#### EARS 验收要求

1. 当达人进入报名详情页时，系统应展示该条报名的最新状态和历史进展。
2. 若发布方已向该达人开放联系方式，系统应在报名详情页展示发布方联系方式。
3. 若该报名仍处于可撤回状态，系统应允许达人在详情页执行撤回操作。

### 11.13 消息中心

#### 功能目标

承接平台关键通知，减少用户错过重要状态。

#### 消息类型

- 通告审核结果
- 新报名通知
- 报名状态变更
- 举报处理结果
- 平台规则通知

#### 规则说明

1. 按时间倒序展示。
2. 未读消息显示红点。
3. 点击消息进入对应详情页。

#### EARS 验收要求

1. 当审核状态或报名状态发生变化时，系统应生成对应站内通知。
2. 当用户点击消息时，系统应跳转至相关页面。

### 11.14 举报与黑名单

#### 功能目标

建立最小可行信任体系。

#### 举报对象

- 通告
- 发布方
- 达人

#### 举报原因

- 虚假合作
- 垃圾广告
- 联系方式异常
- 违规内容
- 骚扰或欺诈
- 其他

#### 平台处置动作

- 警告
- 下架通告
- 限制发布
- 限制报名
- 拉黑账号

#### EARS 验收要求

1. 当用户发起举报时，系统应记录举报对象、原因和补充说明。
2. 若同一对象被多次有效举报，系统应进入人工复核。
3. 若账号进入黑名单，系统应禁止其继续发布或报名。

### 11.15 举报记录

#### 功能目标

让用户追踪自己已提交举报的处理进展，减少“举报后石沉大海”的感受。

#### 详细需求

1. 展示用户本人提交的举报记录列表。
2. 每条记录至少展示举报对象、举报原因、提交时间、当前处理状态。
3. 若举报已处理，支持查看处理结论摘要。
4. 支持从消息中心跳转到对应举报记录。

#### EARS 验收要求

1. 当用户进入举报记录页时，系统应展示其历史举报记录及最新处理状态。
2. 若举报已有处理结果，系统应向用户展示结论摘要。

### 11.16 意见反馈

#### 功能目标

为用户提供低门槛的问题反馈和产品建议入口。

#### 详细需求

1. 支持选择反馈类型，例如功能建议、问题反馈、体验吐槽、其他。
2. 支持填写问题描述。
3. 支持上传截图。
4. 支持填写回访联系方式，默认为可选。
5. 提交成功后展示确认反馈文案。

#### EARS 验收要求

1. 当用户提交意见反馈时，系统应记录反馈类型、描述和时间。
2. 若用户未填写问题描述，系统应阻止提交并提示补充。

### 11.17 我的页面

#### 功能目标

集中承载账户资产与设置入口。

#### 页面内容

- 头像与昵称
- 当前身份
- 资料完整度
- 我的通告
- 我的报名
- 达人名片
- 规则说明
- 意见反馈
- 举报记录
- 双角色优先视角切换

#### EARS 验收要求

1. 系统应在“我的”页清晰展示用户当前可执行的核心入口。
2. 当双角色用户切换优先视角时，系统应仅改变头部摘要和入口排序，不改变真实权限和已有数据。

## 12. 最小可行风控方案

### 12.1 风控目标

在不要求企业认证的情况下，把风险控制在可接受范围内。

### 12.2 风控手段

1. 新用户限量发布：新用户每日最多发布 2 条通告。
2. 审核前置：新用户前 3 条必须人工审核。
3. 敏感词规则：命中违规词自动拦截或进入复核。
4. 重复内容检测：高相似度内容进入复核。
5. 举报闭环：所有举报必须可追踪处理结果。
6. 黑名单机制：多次违规用户加入黑名单。
7. 资料完整度提示：信息缺失严重的通告降低曝光或不予通过。

### 12.3 风控分级

- 低风险：正常审核通过
- 中风险：人工复核后上架
- 高风险：驳回或直接下架

## 14. 广告变现策略

### 14.1 变现原则

1. V1 不接入影响核心转化的广告位。
2. 广告不能打断“发布”和“报名”链路。
3. 广告优先放在弱干扰场景，而非高意图操作场景。

### 14.2 后续可接入广告位

- 广场信息流广告
- 列表底部广告位
- 我的页 Banner
- 规则页或活动页广告位

### 14.3 广告接入前置条件

1. 日活和通告供给达到基本规模。
2. 核心链路转化率稳定。
3. 举报率和跳失率处于可控水平。

## 15. 第一期明确不做

1. 企业认证与营业执照强校验
2. 会员体系
3. 付费置顶
4. 超级报名
5. 平台担保
6. 平台结算
7. 站内即时聊天
8. 自动同步第三方平台账号数据
9. 复杂推荐算法
10. Excel 导出

## 16. 风险与对策

### 16.1 风险

1. 低门槛导致垃圾内容增加
2. 虚假发布方影响达人信任
3. 免费策略导致早期营收为空
4. 达人和发布方两侧冷启动速度不一致

### 16.2 对策

1. 用审核、举报、黑名单做基础风控
2. 用资料完整度、历史发布数、历史违规记录做轻信任标签
3. 先不追求变现，优先验证撮合效率
4. 优先切一个垂直场景做冷启动，例如本地探店

## 18. 附录：关键需求清单（EARS 汇总）

1. 系统应允许用户在不完成企业认证的前提下浏览、发布和报名。
2. 当用户首次点击发布时，系统应要求其补充发布方最小资料。
3. 当用户首次点击报名时，系统应要求其补充达人最小资料。
4. 当用户提交通告时，系统应校验必填项并将其送入审核流程。
5. 当新用户提交前 3 条通告时，系统应进入人工审核。
6. 若通告命中敏感词、重复发布或联系方式异常，系统应进入人工复核。
7. 当通告审核通过时，系统应在广场展示该通告。
8. 当达人查看通告详情页时，系统应展示报名决策所需的完整信息。
9. 若通告已截止、已关闭或已下架，系统应禁止继续报名。
10. 当达人提交报名时，系统应避免对同一通告重复报名。
11. 当发布方收到新报名时，系统应发送站内通知。
12. 当报名状态发生变化时，系统应同步通知达人。
13. 当用户发起举报时，系统应记录举报内容并进入处理流程。
14. 若账号被拉入黑名单，系统应禁止其继续发布和报名。
15. 系统应保持核心能力免费，不通过会员或付费权限阻断核心链路。
16. 当发布方查看报名联系方式时，系统应仅对该通告对应报名展示完整联系方式并记录查看行为。
17. 当发布方将报名标记为待联系时，系统应向对应达人开放发布方联系方式。
18. 当通告被转人工复核时，系统应保持前台状态为待审核并创建新的复核任务。
