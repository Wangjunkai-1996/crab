- 文档日期：2026-03-15
- 文档目标：区分通告业务状态和审核任务状态，补齐“转人工复核”后的队列和任务生命周期

## 2. 核心原则

1. 通告业务状态和审核任务状态是两套概念，不能混用。
2. “转人工复核”是审核任务结果，不是前台通告的新业务状态。
3. 一次提交通告可关联多条审核任务，但前台只关心一个通告业务状态。
4. 每次重新提交审核，都应新建一条审核任务，保留历史任务记录。

## 3. 两套状态定义

### 3.1 通告业务状态

| 状态 | 对前台含义 |
| --- | --- |
| 草稿 | 用户填写中，未进入审核 |
| 待审核 | 已提交，平台尚未给出最终业务结果 |
| 驳回待修改 | 不通过，但允许直接改后重提 |
| 需补充资料 | 信息不足，需要补更多材料后重提 |
| 进行中 | 已通过并上架 |
| 已截止 | 到达截止时间 |
| 已关闭 | 发布方主动关闭 |
| 已下架 | 因违规或举报处理而下架 |

### 3.2 审核任务状态

| 状态 | 对后台含义 |
| --- | --- |
| 待处理 | 任务已进入队列，尚未被领取 |
| 处理中 | 审核员已打开并接手处理 |
| 已完成 | 当前任务已给出结果并结束 |
| 已取消 | 任务作废或被系统关闭 |

### 3.3 审核任务阶段

| 阶段 | 说明 |
| --- | --- |
| 初审 | 通告首次提交后的审核 |
| 人工复核 | 命中高风险、争议或规则无法判定时的复核 |
| 补充后复审 | 用户补充资料或修改后再次提交形成的新任务 |

## 4. 审核结果与业务状态映射

| 审核结果 | 当前任务处理 | 通告业务状态变化 | 是否新建下一条任务 |
| --- | --- | --- | --- |
| 通过 | 当前任务完成 | 变为进行中 | 否 |
| 驳回 | 当前任务完成 | 变为驳回待修改 | 否 |
| 需补充资料 | 当前任务完成 | 变为需补充资料 | 否 |
| 转人工复核 | 当前任务完成 | 保持待审核 | 是，创建人工复核任务 |
| 直接下架 | 当前任务完成 | 变为已下架 | 否 |

## 5. 标准流程

### 5.1 首次提交通告

1. 用户提交审核。
2. 通告业务状态进入 `待审核`。
3. 系统创建一条 `初审` 任务，任务状态为 `待处理`。

### 5.2 审核员领取任务

1. 审核员进入任务详情页或主动领取任务。
2. 任务状态切换为 `处理中`。
3. 若 V1 暂不做强锁定，至少要记录 `assignedTo` 和 `claimedAt`。

### 5.3 审核员给出结果

1. 审核员必须填写处理动作、原因分类和处理备注。
2. 若结果是 `通过 / 驳回 / 需补充资料 / 直接下架`，当前任务直接结束。
3. 若结果是 `转人工复核`，当前任务结束，同时自动创建下一条 `人工复核` 任务。

### 5.4 用户修改后重新提交

1. 通告原业务状态为 `驳回待修改` 或 `需补充资料`。
2. 用户修改完成并再次提交。
3. 通告业务状态重新变为 `待审核`。
4. 系统新建一条 `补充后复审` 任务，历史任务保留不覆盖。

## 6. 任务流转表

### 6.1 审核任务状态流转

| 当前任务状态 | 触发动作 | 下一任务状态 |
| --- | --- | --- |
| 待处理 | 审核员领取 | 处理中 |
| 处理中 | 提交审核结果 | 已完成 |
| 处理中 | 释放任务 | 待处理 |
| 待处理 / 处理中 | 系统判定任务失效 | 已取消 |

### 6.2 通告与任务联动流转

| 当前通告状态 | 触发动作 | 下一通告状态 | 任务变化 |
| --- | --- | --- | --- |
| 草稿 | 提交审核 | 待审核 | 创建初审任务 |
| 待审核 | 通过 | 进行中 | 当前任务完成 |
| 待审核 | 驳回 | 驳回待修改 | 当前任务完成 |
| 待审核 | 需补充资料 | 需补充资料 | 当前任务完成 |
| 待审核 | 转人工复核 | 待审核 | 当前任务完成，并创建复核任务 |
| 待审核 | 直接下架 | 已下架 | 当前任务完成 |
| 驳回待修改 | 重新提交 | 待审核 | 创建补充后复审任务 |
| 需补充资料 | 重新提交 | 待审核 | 创建补充后复审任务 |

## 7. 队列设计建议

### 7.1 V1 最小队列

| 队列 | 进入条件 | 处理人 |
| --- | --- | --- |
| 初审队列 | 普通提交通告 | 审核员 |
| 人工复核队列 | 命中敏感词、联系方式异常、历史争议、多人举报等 | 运营管理员或高级审核员 |
| SLA 超时队列 | 待处理时间超过阈值 | 运营管理员 |

### 7.2 排序优先级

1. 高风险通告优先。
2. 被多人举报或与历史违规账号关联的内容优先。
3. 即将超时或已超时的任务优先。

## 8. 前后台联动规则

### 8.1 用户端可感知状态

1. 用户只看到 `待审核 / 驳回待修改 / 需补充资料 / 进行中 / 已截止 / 已关闭 / 已下架` 等业务状态。
2. 用户不看到“初审中”“复核中”等后台任务状态。
3. “转人工复核”不直接通知用户，除非因此导致 SLA 延长且产品另有文案需求。

### 8.2 消息通知触发

| 场景 | 是否通知用户 | 通知对象 |
| --- | --- | --- |
| 审核通过 | 是 | 发布方 |
| 驳回 | 是 | 发布方 |
| 需补充资料 | 是 | 发布方 |
| 转人工复核 | 否 | 无 |
| 直接下架 | 是 | 发布方 |

## 9. 后台必填记录

审核任务提交结果时至少需要记录：

1. 任务 ID
2. 通告 ID
3. 审核阶段
4. 处理动作
5. 原因分类
6. 处理备注
7. 操作人
8. 操作时间
9. 下一队列去向（仅转人工复核时）

## 10. 开发注意事项

1. 前台通告状态不要直接复用后台任务状态字段。
2. 任务流转和通告流转都应写操作日志，便于申诉、举报和风控排查。
3. 若 V1 不做任务锁定，也要避免两个审核员重复提交同一任务结果。
